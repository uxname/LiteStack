# Audit Report: Logging Best Practices — 2026-06-20 21:47

Проект: Go-бэкенд (`github.com/uxname/liteend-go`), логирование на `slog`.
Baseline: отсутствует (`docs/audit-baseline.yml` не найден) — записей `ACCEPTED` нет.

## Кратко про инфраструктуру логирования

- Единый конструктор `logger.New(level)` всегда отдаёт JSON-логгер (`slog.NewJSONHandler` в `os.Stdout`) — структурированный формат одинаков для dev и prod (`internal/logger/logger.go:26-32`).
- В request scope логгер берётся из контекста `logger.From(ctx)` и несёт `request_id` (добавляется в `internal/middleware/logging.go:21`) и `user_id` (добавляется в `internal/auth/middleware.go:19`).
- Для фоновых/lifecycle-путей используется инжектированный `*slog.Logger`.
- Чувствительные ключи редактируются на уровне хендлера через `ReplaceAttr` (`internal/logger/logger.go:47-52`) и отдельно для GraphQL-переменных (`internal/graph/logging.go:57-70`). Список ключей: `password, token, secret, authorization, credentials, cookie, sig`.

## Дополнительные наблюдения (вне строгого чеклиста)

- Горячих циклов с логированием на каждую итерацию не обнаружено. Логи в очереди привязаны к жизненному циклу задачи (старт/финиш/ошибка), а не к внутренним циклам (`internal/queue/queue.go:137-149`). Удаление старых бэкапов логирует по элементу (`internal/backup/backup.go:165-168`), но число файлов ограничено политикой ретеншена — не «горячий» путь.
- Стек-трейсы пишутся только при паниках на уровне `Error` (`internal/middleware/recover.go:27`, `internal/queue/queue.go:123`) — это оправдано, не избыточно.
- Уровни логирования расставлены корректно: `Info` для нормальных событий, `Warn` для деградаций (промах кэша, пропуск бэкапа), `Error` для сбоев, `Debug` для дедупликации/диагностики.
- Редактирование чувствительных ключей продублировано в двух местах (`internal/logger/logger.go:12-20` и `internal/graph/logging.go:14-17`) с одинаковым списком — риск рассинхронизации при добавлении нового ключа. Стоит вынести список в один пакет.
